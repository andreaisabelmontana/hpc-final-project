What it is

Making a real workload fly

The whole project hangs on one discipline: measure before you optimise. It starts with a working but slow program, profiles it to find where the time actually goes, and then attacks that hotspot with parallelism — shared-memory threads within a node, message passing across nodes, and the scheduling glue to run it all on a cluster.

What makes it a proper capstone is that it ends in evidence. Not "it feels faster" but scaling curves: how runtime falls as you add cores, where the speedup plateaus, and why. That gap between ideal and actual speedup is the entire lesson of high-performance computing.

Architecture

How the speedup is earned

The work follows the cardinal HPC discipline — measure, change one thing, measure again:

  1. Baseline

    Get a correct serial version and time it as the reference point.

  2. Profile

    Locate the hotspot consuming most of the runtime.

  3. Parallelise

    Apply OpenMP within nodes and MPI across them, targeting that hotspot.

  4. Scale

    Run on the cluster via Slurm at increasing core counts.

  5. Report

    Produce scaling curves and explain the gap to ideal speedup.

Reflection

What rebuilding it taught me

  • Profile first, always. The hotspot is never where you guessed; optimising the wrong loop is wasted effort.
  • Amdahl's law is undefeated. The serial fraction you can't parallelise sets a hard ceiling — no number of cores breaks it.
  • Communication is the tax. Across nodes, the cost of moving data often eats the gain from splitting the work.
  • Correctness comes first. A parallel bug — a race condition — produces a fast, confidently wrong answer.
  • Evidence beats intuition. The scaling curve, not a stopwatch feeling, is what tells you whether you actually won.
