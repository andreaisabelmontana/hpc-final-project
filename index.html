Mandelbrot, parallelised and measured

Take one genuinely compute-heavy workload — escape-time rendering of the Mandelbrot set — and make it fly: profile the hotspot, vectorise it with NumPy, spread it across CPU cores with multiprocessing, and then prove the speedup with a real strong-scaling study. All four implementations return the same pixels; every speedup is a measured win, not a changed answer.

The output

What it computes

For every pixel c of a complex-plane grid, iterate z → z² + c until |z| > 2 and record how many steps it took. That iteration count is the image. The workload is embarrassingly parallel — pixels are independent — but the per-pixel cost is wildly uneven (points inside the set run the full budget), which is what makes the scaling honest rather than trivially linear.

The implementations

Four ways to render the same image

Same input, same output array (verified), increasing speed.

reference

naive

A pure-Python double loop over pixels. Correct and readable, and the baseline everything is timed and checked against.

vectorised

numpy

One NumPy pass over the whole grid — Python-interpreter overhead paid once per iteration, not once per pixel.

parallel

multiprocessing

Split the rows into bands, run the NumPy kernel per band in its own process, stitch the bands back together.

optional JIT

numba

An @njit(parallel=True) kernel, enabled automatically if numba is installed. In-process threads, no spawn cost.

The vectorised kernel, up close

How you vectorise an algorithm with a variable trip count

Vectorising a fixed amount of arithmetic is easy. Mandelbrot is harder: different pixels escape at different times, so a plain "do max_iter passes over the whole grid" would keep iterating points that have already left the disc — and worse, those runaway values would not match the naive reference, which stops counting the moment a pixel escapes. The kernel solves this with a boolean escape mask instead of per-pixel branching.

zr = zi = out = zeros(grid)            # real, imag, iteration count
for _ in range(max_iter):
    mag2  = zr*zr + zi*zi
    alive = mag2 <= 4.0                 # pixels still inside |z| ≤ 2
    if not alive.any(): break          # everyone escaped → stop early
    out[alive] += 1                    # count one more step, survivors only
    new_zr = zr*zr - zi*zi + cr        # z = z² + c, on the whole grid
    new_zi = 2*zr*zi      + ci
    zr = where(alive, new_zr, zr)      # freeze escaped pixels in place
    zi = where(alive, new_zi, zi)

Three details make this both fast and bit-for-bit identical to the pure-Python loop:

  1. The mask replaces the branch

    alive = mag2 <= 4.0 is a full-grid boolean array. out[alive] += 1 increments only the pixels that began this pass still bounded — mirroring exactly when the naive loop's count += 1 runs, so the integer counts agree everywhere.

  2. Freeze, don't branch out

    NumPy can't skip escaped pixels — the array math runs over the whole grid every pass. So np.where(alive, new_z, z) keeps escaped pixels frozen at their last value: their result is already locked in, and freezing also dodges overflow warnings from z² on runaway points.

  3. Cheap early exit

    Once alive.any() is false every pixel has escaped, so the loop breaks before reaching max_iter — most views never run the full budget.

This is the heart of why the speedups on this page are honest: the test suite asserts the vectorised and parallel kernels reproduce the naive array exactly (iteration counts are integers, so it's exact equality, not a tolerance). The same _mandelbrot_numpy_block is what each multiprocessing worker runs on its slab of rows — so parallel and serial share one code path and one correctness proof.

The parallel kernel, up close

How the work is split across processes

Because every pixel is independent, the image needs no clever partitioning — any disjoint cover of the rows works. The kernel takes the simplest one: cut the grid into horizontal row-bands, hand each band to its own process, and stack the answers back. np.array_split(np.arange(height), workers) produces near-equal contiguous bands — when the rows don't divide evenly the band sizes differ by at most one row — so an 800-row image over 16 workers becomes sixteen 50-row slabs, each a self-contained sub-render.

bands = np.array_split(np.arange(height), workers)   # near-equal row slabs
tasks = [(xs, ys[band], max_iter) for band in bands]
ctx   = mp.get_context("spawn")               # portable start method (Windows)
with ctx.Pool(processes=workers) as pool:
    results = pool.map(_worker_rows, tasks)          # one band per process
return np.vstack(results)                            # re-stack, top to bottom

A few details are what make this a clean, correct parallel decomposition rather than just "run it on more cores":

  1. Each band carries its own slice of the plane

    A worker receives the shared real axis xs but only its rows of the imaginary axis, ys[band]. It runs the exact same _mandelbrot_numpy_block the serial path uses — there is no separate parallel maths to get wrong, which is why parallel output is bit-for-bit identical to naive.

  2. No communication during compute

    Rows never need each other's values, so once the bands are dealt out there is zero inter-process traffic until the end. The only data that crosses a process boundary is each band's finished result on the way back — the textbook embarrassingly-parallel shape.

  3. Picklable by construction

    _worker_rows is a module-level function, not a closure or lambda, so the spawn start method (the only portable one on Windows) can import and call it in a fresh interpreter. Each band's inputs and result are plain NumPy arrays, which pickle cleanly across the pipe.

  4. Order-preserving reassembly

    pool.map returns results in task order regardless of which process finishes first, so np.vstack rebuilds the image top-to-bottom with no bookkeeping. A dedicated test renders with four workers and asserts the stacked array equals the serial one — catching any band that lands in the wrong place.

One deliberate shortcut: at workers == 1 the kernel skips the pool entirely and calls the block function directly. That makes the single-worker time in the scaling table a true serial baseline — it pays none of the spawn-and-pickle tax, so the speedup ratios measure real parallel gain rather than flattering themselves against a pool-of-one.

The evidence

Strong-scaling study — real numbers

This is a strong-scaling study: the problem size is held fixed (800×800, max_iter=1000) while the worker count grows — asking "given this exact image, how much faster do more cores make it?" (The complement, weak scaling, would grow the image with the cores to keep per-worker work constant; that's not what's measured here.) Each point is the best of 3 runs after a warmup. Measured on a 16-core machine. speedup = T(1)/T(p), efficiency = speedup/p — ideal speedup is p, ideal efficiency 100%.

Read honestly. The speedup is real (2.76x on 16 cores) but plainly sub-linear, and efficiency collapses as cores are added. That's Amdahl's law: the fixed overhead of this setup — spawning processes, pickling each result band back, the final concatenation — is work a fast NumPy kernel finishes in seconds, so more processes hit diminishing returns fast. On Windows the only portable start method is spawn, whose per-process cost is high, which is why the knee comes early.

Amdahl's law, made concrete. If a fraction s of the work is inherently serial, the best speedup on p workers is S(p) = 1 / (s + (1 − s)/p) — and as p → ∞ it tops out at 1/s, no matter how many cores you add. This study doesn't measure s directly, so as a purely illustrative number: if the serial part were 5% of the work, 16 cores would yield at most 1/(0.05 + 0.95/16) ≈ 9.1×, with a hard ceiling of 1/0.05 = 20×. The measured 2.76× sitting below even that hypothetical says the real serial-plus-overhead share here is larger still — exactly the diminishing-returns shape the curve shows.

The honest punchline: vectorisation is the big win. At this size the multiprocessing version barely matches single-process NumPy — spawn overhead eats the gain — and only pulls ahead on the larger scaling instance above. Numba's in-process JIT avoids all process overhead and wins by three orders of magnitude: the right tool when you can use it.

Scope

Single node, honestly — and how it reaches MPI

This is a single-node, multi-core study. A real cluster with many machines and MPI is out of scope for one laptop — so the page doesn't claim it. But the decomposition used here is the distributed one, and the jump is mechanical:

  1. Domain decomposition

    Give MPI rank r of n its slice of rows — the same split this code already does.

  2. No halo exchange

    Pixels are independent, so unlike a stencil or PDE solver there is zero communication during compute — only the final gather.

  3. Gather

    Replace the process-pool map and concatenate with an MPI Gatherv onto rank 0.

  4. The real ceiling

    Communication is cheap (one gather); load imbalance isn't — an equal-rows split hands some ranks mostly-interior, expensive rows. A serious version would deal out smaller chunks dynamically.
